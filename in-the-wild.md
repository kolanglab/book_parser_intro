# 言語処理系の現場のパーサ

ここまで理論から最先端まで一通り見てきました。最後に、抽象を離れて現実を見ましょう ── みなさんが毎日使っているプログラミング言語の処理系は、実際のところ**どんなパーサを使っている**のでしょうか。そして、その処理系は自分のパーサを**外部から使える形で提供している**のでしょうか。本章では主要な言語処理系を巡り、本書で学んだ手法がどう実地に投入されているかを確かめます。理論で予想したことと現場の選択が一致する場面も、意外な逆転を見せる場面もあります。

## 意外な事実：本物のコンパイラは「手書き」が多い

最初に、おそらく本書で最も意外な事実をお伝えします。第4章であれほど「パーサージェネレータは大規模文法に強い」と説いたにもかかわらず、**世界の主要な言語処理系の多くは、パーサを手書きの再帰下降で実装しています**。第3章で作ったあの素朴な手法が、最前線でも主役なのです。

具体例を挙げましょう。

- **C / C++（Clang）**：完全に手書きの再帰下降パーサ。
- **C / C++（GCC）**：かつては Bison 製の文法を使っていましたが、2004年の GCC 3.4 で C++ フロントエンドを**手書きの再帰下降に置き換え**ました。
- **Go**：標準コンパイラのパーサは手書きの再帰下降。
- **Rust（rustc）**：手書きの再帰下降。
- **JavaScript（V8 など）**：主要エンジンのパーサは手書きの再帰下降。式の解析には第3章末で触れた Pratt 法系の手法を使います。
- **Java（javac）**：手書きの再帰下降。
- **C#（Roslyn）**：手書きの再帰下降。

なぜ理論的に「強い」はずのジェネレータより、素朴な手書きが選ばれるのでしょうか。理由は第3章で挙げた手書きの利点に尽きます。第一に **エラーメッセージの質**です。本章の後半で見るように、現代の言語は「親切な診断」を競っており、手書きならエラー回復とメッセージを好きなだけ作り込めます。第二に **速度と制御**です。生成された表ベースのパーサより、専用に書いたコードのほうが速く、メモリ配置も制御できます。第三に **柔軟性**です。実在の言語には、文脈依存の厄介な構文（型と式の見分け、改行が意味を持つ規則など）が必ずあり、ジェネレータの枠に収まりきらない部分を手書きなら力ずくで処理できます。

> [!IMPORTANT]
> ここから引き出すべき教訓は「ジェネレータは役立たず」ではありません。第3章で書いた小さな再帰下降は、**おもちゃではなく本物の技術**だった、ということです。基礎をしっかり理解した手書きパーサは、最も成熟した処理系で今も第一線を張っています。一方、文法が頻繁に変わる試作段階や DSL では、次に見るようにジェネレータが圧倒的に効率的です。適材適所なのです。

## Ruby の二本立て：手書き Prism と自作ジェネレータ Lrama

第3章で見た手書きの再帰下降が主要処理系で主役を張る一方、パーサージェネレータが廃れたわけでは決してありません。両者が同じ言語の中で**並び立っている**、ちょっと変わった現場すらあります。その象徴が **Ruby** です。

Ruby は長らく、yacc／Bison 系の道具で `parse.y` という文法ファイルから LALR パーサを生成してきました。本書で学んだ LR 理論[](#cite:knuth1965)が、世界中で動く Ruby インタプリタの心臓部に直接効いていたわけです。とはいえ Ruby の文法は文脈依存の難所が多く、`parse.y` は巨大で複雑なことで知られ、長年「触るのが難しいファイル」の代名詞でした。加えて、生成に使う Bison のバージョンが環境ごとに違うことが、ビルドの悩みの種にもなっていました。

近年、この状況に二方向からの変化が起きます。一つは **Prism** という新しい手書きの再帰下降パーサの登場です[](#cite:prism)。式の解析には第3章末で触れた Pratt 法を採用し、エラー回復に強く、C と Ruby のライブラリとして多言語から再利用しやすいことを目標に設計されました。もう一つは、**ジェネレータの路線を捨てずに自前で作り直した**ことです。2023年の Ruby 3.3 で、`parse.y` を処理する道具は Bison から **Lrama** へ置き換えられました[](#cite:lrama)。Lrama は Ruby で書かれた純粋な LALR パーサージェネレータで、Bison 互換から出発しつつ Ruby 独自の機能拡張を重ねており、これにより外部の Bison への依存も解消されています。

結果として Ruby は、**手書きの Prism と、ジェネレータ Lrama が生成する `parse.y` ベースのパーサという、2つのパーサを並行して保守する**という珍しい状態にあります。Ruby 3.4（2024年）で既定のパーサは Prism に切り替わりましたが、安定性と互換性のために `parse.y` と Lrama の経路も引き続き維持されています。現状は「開発者向け API と将来の共通基盤として Prism を中心に据えつつ、既存の `parse.y` / Lrama 経路も並行して保守する」段階と見るのがよいでしょう。手書きへ一方通行で移った GCC とは異なり、Ruby はジェネレータという選択肢を生かしたまま、両輪で前に進もうとしているのです。

> [!NOTE]
> 注目したいのは、これが「ジェネレータか手書きか」という単純な二者択一ではない点です。Matz は「今後 Ruby の公式パーサ API は Prism の API とする」と表明しており、CRuby が内部でどちらのパーサを採用しても開発者向けの API は Prism に揃える、という方針が示されています。さらに一歩進めて **Lrama が生成するパーサに Prism と同じ API を持たせ、両者を統合してはどうか**、という構想も語られています。ジェネレータの強み（文法定義を一箇所で管理できること）と手書きの強み（使いやすい API・きめ細かいエラー回復）は、必ずしも対立しないのです。

そして、ジェネレータが最も輝くもう一つの場面が、**DSL（ドメイン特化言語）や言語の試作**です。第5章で見た **ANTLR** は、SQL 方言、設定言語、データ記述言語、各種の社内ツール言語など、無数の DSL のパーサを支えています[](#cite:parr2013)。「文法を書けば数分で動くパーサが手に入る」という生産性は、頻繁に文法が変わる開発初期や、一品物の言語処理ツールでは何物にも代えがたいのです。

## Python の大転換：LL(1) から PEG へ

理論と現場の交差点として、ひときわ象徴的なのが Python です。Python は長年、自前のパーサージェネレータ（pgen）が生成する **LL(1)** パーサを使ってきました。ところが LL(1) は前章で見たとおり「先読み1個で規則を区別できる」文法しか扱えず、表現力に強い制約があります。Python の文法が成長するにつれ、この制約が言語設計の足枷になっていきました ── 書きたい構文が LL(1) で表せないために、文法を不自然にねじ曲げる必要が出てきたのです。

そこで Python は、バージョン3.9（2020年）で内部パーサを **PEG** ベースのものへ全面的に置き換えました。この決定は PEP 617 として文書化されています[](#cite:pep617)。第5章で学んだ PEG[](#cite:ford2004)とパックラット構文解析[](#cite:ford2002)が、世界で最も使われる言語の一つの中核に採用されたわけです。PEG への移行により、Python は LL(1) の制約を意識して文法を不自然に変形する必要が小さくなりました。たとえば後の版で導入された構造的パターンマッチングのような構文拡張も、より直接的な文法として記述しやすくなったと考えられます（もちろん、こうした言語設計には構文解析以外の要因も関わります）。

Python のこの転換は、本書のテーマを見事に体現しています ── **パーサの選択は、言語そのものの設計可能性を左右する**。どんな文法クラスを扱えるパーサを使うかが、言語デザイナーが描ける構文の限界を決めるのです。パース技術は決して「裏方の地味な実装詳細」ではなく、言語の未来を形づくる選択なのです。

## パーサは「提供」されているか ── ツールのための解析

最後に、現場のもう一つの重要な側面を見ます。言語処理系のパーサは、コンパイラの内部で使われるだけでなく、**外部のツールから再利用できる形で提供されているか**という問題です。リンタ、フォーマッタ、IDE、ドキュメント生成器、コード変換ツール ── これらはみな、対象言語のパーサを必要とします。各言語が「自分のパーサを部品として開放」しているかどうかで、その言語のツール生態系の豊かさが決まります。

多くの現代言語は、パーサを公式に提供しています。

- **Python**：標準ライブラリの `ast` モジュールで、ソースを AST に変換し、また AST を操作できます。
- **Ruby**：`RubyVM::AbstractSyntaxTree` に加え、前述の Prism がライブラリとして AST を提供し、多言語から呼べるよう設計されています。
- **Go**：標準ライブラリ `go/parser`・`go/ast` が、コンパイラと同じ解析を外部に開放しています。`gofmt` をはじめ Go の充実したツール群は、これに支えられています。
- **C#（Roslyn）**：コンパイラ自身が API として構文木を公開する「コンパイラ・アズ・ア・サービス」を掲げ、IDE と分析ツールの基盤になっています。
- **C / C++（Clang）**：`libclang` などを通じて AST へアクセスでき、多くの解析ツールが Clang のフロントエンドを再利用しています。

そして、言語の壁を越えて「あらゆる言語のパーサを一つの枠組みで提供する」試みが、第5章・第6章で見た **tree-sitter** です[](#cite:treesitter)。tree-sitter は数百の言語の文法を共通の仕組みで提供し、エディタや解析ツールが「言語ごとに別々のパーサを抱える」苦労から解放されつつあります。一つのインクリメンタルな解析基盤の上に、多言語の構文ハイライトや構造解析が載る ── これは「パーサの提供」という観点での、近年最も影響の大きい動きと言えるでしょう。

> [!NOTE]
> パーサが提供されているかどうかは、その言語で**メタプログラミング**やツール開発がどれだけ容易かに直結します。Rust のマクロが構文木を直接扱えるのも、Go のツール群が豊かなのも、根底に「再利用できるパーサ」があるからです。言語を選ぶとき、構文の美しさだけでなく「パーサが開かれているか」にも目を向けると、その言語の生態系の底力が見えてきます。

## 本書のまとめ ── テキストから木へ、そしてその先へ

長い旅が終わりに近づきました。本書をたどり直してみましょう。

第1章では、なぜパースするのか ── プログラムの再帰的な構造を扱い、後段の全処理の土台となる AST を作るため ── を確認しました。第2章で文法・導出・構文木という共通言語を整え、第3章では自分の手で再帰下降パーサを書き、文法がコードになる手応えを掴みました。第4章・第5章で Bison から ANTLR、PEG、コンビネータ、tree-sitter まで多彩な道具を巡り、第6章でその裏にある LL・LR・PEG・Earley の理論へ降り、第7章で GLR・GLL・微分・インクリメンタル・エラー回復という最前線を訪ねました。そして本章で、それらが現実の処理系でどう生きているかを見届けました。

入口の電卓は素朴でしたが、行き着いた先は、半世紀以上の理論研究と、世界中の処理系開発者の工夫が積み重なった豊かな世界でした。構文解析は「枯れた基礎技術」どころか、エディタ支援や言語設計の最前線で今も進化を続けています。本書はあくまで入口です。ここから先、より深く学びたい読者には、本書が随所で引用してきた古典 ── ドラゴンブック[](#cite:aho2006)、Cooper と Torczon の教科書[](#cite:cooper2011)、そして構文解析を網羅的に扱った Grune と Jacobs の大著[](#cite:grune2008) ── が確かな道しるべになるでしょう。

テキストから木を育てる技術は、これからも進化し続けます。その木の上で、あなたが次にどんな道具を作るのか ── それを楽しみにして、本書を閉じます。

## まとめ

- 主要な言語処理系（Clang, GCC, Go, Rust, V8, javac, Roslyn）の多くは**手書きの再帰下降**を採用する。第3章の手法は最前線の現役技術である。
- **ジェネレータ**は DSL や試作で圧倒的に効率的なだけでなく、主要言語の中核でも現役。Ruby は手書きの **Prism**（3.4 で既定）と、Bison を自作の LALR ジェネレータ **Lrama** に置き換えた `parse.y` 経路という、2つのパーサを並行して保守している。
- **Python は LL(1) から PEG へ**転換し（PEP 617）、パーサの選択が言語の設計可能性を左右することを示した。
- 現代言語はパーサを**ライブラリとして提供**し、ツール生態系を支える。**tree-sitter** は多言語パーサを単一の枠組みで提供する。
- 構文解析は古典でありながら、今も最前線で進化を続ける生きた技術である。
