# 様々なパーサージェネレータ

前章の Bison は LR 系ジェネレータの代表選手でしたが、構文解析ツールの世界はそれだけではありません。トップダウンを自動化したもの、まったく別の文法形式に基づくもの、ホスト言語の関数として文法を組み立てるもの、編集中のコードを高速に追従するもの ── それぞれが異なる設計思想を持ち、得意分野も使い心地も大きく異なります。本章では代表的な4系統を巡り、「なぜこんなに方式があるのか」を体感します。ここで各ツールの**性格**を掴んでおくと、次章の理論が「どのツールがどの理論に対応するのか」という地図として頭に入ります。

> [!NOTE]
> 本書では、LL・LR・PEG・GLR といった**理論上の分類**と、Bison・ANTLR・tree-sitter・Prism といった**実用システム**を同じ流れで扱います。ただし実用システムは複数の技法を組み合わせて作られており、教科書的な分類にぴったり収まるとは限りません。分類名は理解のための地図であり、実装そのものではない、という点は頭の片隅に置いておいてください。

## ANTLR ── 賢いトップダウン

**ANTLR（ANother Tool for Language Recognition）** は、Terence Parr が長年開発を続けているトップダウン系のジェネレータで、Java をはじめ多くの言語のターゲットコードを生成できます。Bison がボトムアップ（LR）なのに対し、ANTLR は前章の手書き再帰下降を**自動生成する**道具だと考えると分かりやすいでしょう。実際、ANTLR が吐き出すのは人間にも読める再帰下降パーサです。

トップダウンには「左再帰が書けない」「先読みが効かないと選択を誤る」という弱点がありましたが、ANTLR 4 はこれを **ALL(\*)（Adaptive LL(\*)）** という手法で大きく克服しました[](#cite:parr2014)。ALL(\*) の鍵は、どの規則を選ぶかを**実行時に動的に決める**点にあります。従来のトップダウンが「文法を見て静的に先読み表を作る」のに対し、ALL(\*) は実際の入力に対してミニ・パースを走らせ、どの選択肢が成功しそうかを動的解析で見極めます。これにより、固定長の先読みでは区別できない文法も扱え、さらに**左再帰を自動で書き換えてくれる**ため、前章で私たちが手作業でやった変換が不要になります。

文法ファイルは直感的です。四則演算ならこう書けます。

```antlr
grammar Calc;

expr : expr ('*'|'/') expr   # MulDiv   // 左再帰をそのまま書ける
     | expr ('+'|'-') expr   # AddSub
     | INT                   # Int
     | '(' expr ')'          # Paren
     ;

INT  : [0-9]+ ;
WS   : [ \t\r\n]+ -> skip ;   // 空白は字句解析段階で捨てる
```

`expr` が自分自身を左端で参照していますが ANTLR は問題なく処理します。さらに ANTLR 4 は、直接左再帰を含む式規則を特別に解析し、左再帰している選択肢の並び順から演算子の優先順位を導きます。この例では `*`/`/` の選択肢を `+`/`-` より前に置いたので、掛け算・割り算のほうが高い優先順位になります（これは直接左再帰の規則に対する機能で、あらゆる選択肢について「上に書いたものが常に高優先順位」という意味ではありません）。`#MulDiv` のようなラベルを付けておくと、生成されたパーサに対して**訪問者（visitor）**や**リスナー（listener）**のメソッドが自動で用意され、AST を歩いて処理を書けます。文法と処理コードを分離できるこの設計は、大規模言語のツール開発で重宝されています。代償として、ALL(\*) は実行時に動的解析を行うぶん、最悪計算量は LR より大きくなりえますが、実際の文法では十分高速に動くよう工夫されています。

> [!TIP]
> ANTLR は「文法だけ書けば、あとはホスト言語のオブジェクトとして木が手に入る」体験の良さで人気があります。エラー回復機能も組み込みで充実しており、IDE の構文ハイライトや補完のような「壊れたコードを相手にする」用途にも向いています。Parr 自身による解説書[](#cite:parr2013)が定番の入門書です。

## PEG ── 順序付き選択で構文解析を定義するもう一つの文法形式

前章で何度も顔を出した「曖昧性」や「コンフリクト」は、文脈自由文法（CFG）が本質的に抱える厄介ごとでした。これに対し、Bryan Ford が2004年に提案した **PEG（Parsing Expression Grammar, 解析表現文法）** は、そもそも曖昧性が生じない文法形式です[](#cite:ford2004)。

CFG と PEG の決定的な違いは、選択 `|` の意味にあります。CFG の `A | B` は「A でも B でもよい（どちらの導出もありうる）」という**対称な選択**で、これが曖昧性の源でした。PEG の選択は `/` と書き、これは **順序付き選択（ordered choice）** です。`A / B` は「まず A を試し、成功したらそれで確定。失敗したときだけ B を試す」という**非対称**な意味を持ちます。常に左から順に試して最初に成功したものを採るので、解釈は必ず一通りに決まり、**曖昧性が原理的に存在しません**。前章の「ぶら下がり else」も、PEG では「先に書いたほうが勝つ」と機械的に決着します。

注意したいのは、PEG が CFG の単なる「曖昧でない版」では**ない**ことです。記法の見た目は似ていても選択の意味論が異なるため、同じように見える文法でも CFG と PEG では受理する言語や得られる構文木が変わることがあります。PEG は「どう解析するか」という手続きを文法の意味に含めた、CFG とは別の文法形式だと捉えてください（この対比は次章で理論的に掘り下げます）。

PEG はこの順序付き選択に加え、「次が A であることを消費せず確認する」先読み演算子（`&A`、`!A`）を備え、無制限の先読みとバックトラックを許します。表現力は高く、CFG では書きにくい構文も書けます。たとえば「キーワード `if` の後ろが識別子文字でないこと」を `!` で表現でき、字句と構文の境界を柔軟に扱えます。Python の式文法に似た記法の PEG ツールでは、四則演算はこう書けます。

```peg
Expr   <- Term  (('+' / '-') Term)*
Term   <- Factor (('*' / '/') Factor)*
Factor <- '(' Expr ')' / Number
Number <- [0-9]+
```

PEG を線形時間で実行する代表的な実装技法が **パックラット構文解析（packrat parsing）** です。これも Ford が提案したもので[](#cite:ford2002)、バックトラックの度に同じ位置を何度も解析し直すと指数的に遅くなる問題を、**一度計算した「位置×規則」の結果をすべて記憶（メモ化）して使い回す**ことで線形時間に抑えます。メモ化のために入力長に比例したメモリを食う代わりに、無制限の先読みを線形時間で実現するわけです。

> [!NOTE]
> PEG にも弱点はあります。順序付き選択は便利な反面、規則を書く順序を間違えると「先に書いた短い選択肢が常に勝ってしまい、後ろの長い選択肢に永遠に到達しない」という落とし穴があります。また PEG は**左再帰をそのまま書けません**（無限ループになる）。これを克服する研究も進んでおり、メモ化を工夫して左再帰を扱えるようにした手法が提案されています[](#cite:warth2008)。PEG が「魔法の解決策」ではなく、CFG とは別のトレードオフを持つ選択肢であることを押さえておきましょう。

## パーサーコンビネータ ── 関数を組み合わせて文法を作る

ここまでのツールは「専用の文法ファイルを書いて、専用の生成器にかける」スタイルでした。**パーサーコンビネータ（parser combinator）** はまったく別のアプローチを取ります。専用ファイルもコード生成も使わず、**ホスト言語の普通の関数とその合成だけ**でパーサを組み立てるのです。

基本のアイデアは、「パーサ」を**関数**として表すことです。すなわち「入力（の残り）を受け取り、成功したら『解析結果と消費後の残り入力』を、失敗したらその旨を返す関数」をパーサと定義します。そのうえで、小さなパーサを組み合わせて大きなパーサを作る高階関数 ── これが **コンビネータ** ── を用意します。代表的なコンビネータは次の3つです。

- **連接（sequence）**：「A の後に B」を表す。`digit` の後に `digit`、のように繋ぐ。
- **選択（choice）**：「A または B」を表す。どちらかが成功すればよい。
- **繰り返し（many）**：「A の0回以上の繰り返し」を表す。

この発想を、モナドという関数型プログラミングの概念を使って整理し、広く知らしめたのが Hutton と Meijer の解説です[](#cite:hutton1996)。擬似コードで四則演算の式を書くと、文法の構造がそのまま式になります。

```haskell
-- 各コンビネータ (seq, choice, many) は「パーサを作るパーサ」
expr   = chainLeft term   (symbol "+" `orElse` symbol "-")
term   = chainLeft factor (symbol "*" `orElse` symbol "/")
factor = number `orElse` (symbol "(" `andThen` expr `andThen` symbol ")")
```

`chainLeft` は「左結合の二項演算」を表すコンビネータです。文法 `Expr <- Term (op Term)*` が、ライブラリ関数の呼び出し一行に対応しているのが見て取れます。

パーサーコンビネータの魅力は、**専用ツールが一切要らず、ホスト言語の力をそのまま使える**点です。パーサはただの値なので、変数に入れたり、関数で動的に生成したり、ホスト言語のテストフレームワークでテストしたりが自由自在です。デバッグもステップ実行できます。Haskell の Parsec、Scala の各種ライブラリ、Rust の nom など、関数型・モダン言語のエコシステムで広く使われています。一方で、素朴な実装はバックトラックで遅くなりやすく、左再帰も扱えない（手書き再帰下降と同じ制約を負う）ため、性能やエラー報告には作り手の工夫が要ります。

## tree-sitter ── 編集に追従するインクリメンタルパーサ

最後は毛色の違う一台、**tree-sitter** です。Max Brunsfeld が開発したこのツールは、エディタや IDE での利用を主眼に置いています[](#cite:treesitter)。ここまでのツールが「ファイルを丸ごと一度パースする」前提だったのに対し、tree-sitter が解こうとするのは別の問題です ── **ユーザーが1文字打つたびに、ソース全体を最初からパースし直すのは無駄すぎる**。

tree-sitter の中核は **インクリメンタル構文解析（incremental parsing）** です。編集前の構文木を保持しておき、変更があった箇所**だけ**を解析し直して木を更新します。1万行のファイルの1行を直しても、再解析は実質その周辺で済むため、キー入力のたびに木を更新してもエディタが固まりません。この考え方は Wagner と Graham のインクリメンタルLR構文解析の研究[](#cite:wagner1998)に基づいています。

加えて tree-sitter は、エディタが必ず直面する「文法的に不完全・不正なコード」を相手にできるよう、頑強な**エラー回復**を備えています。書きかけの `if (x` のような壊れた入力に対しても、可能な限りもっともらしい木を返します。内部は LR 系の表駆動パーサを基盤に、競合する箇所では複数の解析を並行して保持する、次章で扱う **GLR 構文解析** 的な仕組みを組み合わせており、曖昧さや競合に強い解析を実現しています。教科書的な GLR そのものというより、インクリメンタル更新・エラー回復・具象構文木の保持などを統合した、エディタ用途に特化した generalized LR 系の実用システムと見るのがよいでしょう。文法は JavaScript の DSL で記述します。

```javascript
module.exports = grammar({
  name: 'calc',
  rules: {
    expr: $ => choice(
      prec.left(1, seq($.expr, '+', $.expr)),   // prec で優先順位を数値指定
      prec.left(2, seq($.expr, '*', $.expr)),
      $.number,
      seq('(', $.expr, ')')
    ),
    number: $ => /\d+/
  }
});
```

`prec.left(1, ...)` のように優先順位を数値で与える点は Bison の `%left` と似ていますが、tree-sitter はこれを「壊れたコードでも止まらず、編集に即応する」という目的のために使います。VS Code をはじめ多くのエディタの構文ハイライトや構造解析が、いまや tree-sitter に支えられています。

## どれを選ぶか ── 設計思想の地図

4系統を並べると、それぞれが異なる問いに答えるために生まれたことが分かります。

| 系統 | 代表 | 方式 | 強み | 弱み |
|------|------|------|------|------|
| LR ジェネレータ | yacc / Bison | ボトムアップ | 左再帰OK・大規模文法・曖昧性を早期検出 | コンフリクト対応・エラー報告 |
| 賢いトップダウン | ANTLR | トップダウン (ALL(\*)) | 直感的な文法・左再帰自動処理・visitor | 動的解析のコスト |
| PEG | packrat 系 | トップダウン (順序付き選択) | 曖昧性なし・字句と構文を統合 | 左再帰・選択順序の罠・メモリ |
| コンビネータ | Parsec, nom | トップダウン (関数合成) | ツール不要・ホスト言語の力 | 性能・左再帰・要工夫 |
| インクリメンタル | tree-sitter | LR 系（GLR 的）+ 増分 | 編集追従・エラー耐性 | 実行向きで生成は重め |

選択の指針はおおむねこうです。**実行用の本格的なコンパイラ**なら手書き再帰下降か LR、**新言語の試作や DSL**なら ANTLR や PEG、**関数型のプロジェクト**ならコンビネータ、**エディタ支援ツール**なら tree-sitter、という具合に、解きたい問題が道具を決めます。「最強のパーサージェネレータ」は存在せず、あるのは「この仕事に合った道具」だけなのです。

ここまで、ツールを**使う**側からその性格を眺めてきました。しかし「なぜ LR は左再帰を扱えて、PEG は扱えないのか」「ALL(\*) の『先読み』とは何を見ているのか」といった問いに本当に答えるには、ツールの裏側にある理論へ降りていく必要があります。次章では、いよいよその理論 ── LL・LR・PEG・Earley ── の世界に踏み込みます。

## まとめ

- **ANTLR** はトップダウンを **ALL(\*)** で強化し、左再帰の自動処理と動的解析による賢い選択を実現する。
- **PEG** は**順序付き選択**により曖昧性を原理的に排除し、**パックラット構文解析**で線形時間を達成する。代償は左再帰非対応とメモリ消費。
- **パーサーコンビネータ**は専用ツールなしに、ホスト言語の関数合成だけでパーサを組む。
- **tree-sitter** は**インクリメンタル構文解析**とエラー回復で、エディタの編集に即応する。
- 「最強の道具」はなく、解きたい問題が道具を決める。
